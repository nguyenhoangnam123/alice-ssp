# ADR 0004 — Gateway API instead of legacy Ingress

Status: accepted (MVP1)
Date: 2026-06-01

## Context

The platform exposes tenant services through an internal ALB (VPN-only) and a public ALB
(only for the small set of services explicitly opted in). The user (platform engineer)
asked specifically for "ALB controller with Gateway API" rather than legacy `Ingress`.

The AWS Load Balancer Controller has supported Gateway API since v2.13. Compared to
`Ingress`, Gateway API gives us:

- Cleaner role split: cluster admins own `GatewayClass`/`Gateway`; tenants own `HTTPRoute`.
- Cross-namespace attachment via `ReferenceGrant` / `allowedRoutes` — natural fit for the
  shared-ALB pattern.
- Native traffic-splitting and timeout/retry policies — useful when we add canary releases.
- Single resource model across L4 (TCPRoute, UDPRoute) and L7 (HTTPRoute) so adding
  non-HTTP exposure later is incremental.

## Decision

### Topology

- **Two shared ALBs**, not one-per-tenant: `alb-internal-shared` and `alb-public-shared`,
  both in the cluster-admin namespace `gateway-system`.
- Tenants do **not** own a `Gateway`. They only own `HTTPRoute`s in their own namespace.
- HTTPRoutes attach to the shared Gateways via `parentRefs.namespace=gateway-system`.
- Each shared Gateway's listener uses
  `allowedRoutes.namespaces.from=Selector` with a selector matching the
  `ssp.platform/tenant` label, which the `tenant-namespace` Terraform module always sets.

### Resources

| Layer | Resource | Owned by |
| --- | --- | --- |
| `40-platform-addons` | `GatewayClass` `alb-internal`, `alb-public` | DevOps |
| `40-platform-addons` | `Gateway` `alb-internal-shared`, `alb-public-shared` in `gateway-system` | DevOps |
| `tenant-namespace` module | Tenant `Namespace` with `ssp.platform/tenant=<id>` label | Platform team |
| `helm/app/templates/httproute.yaml` | `HTTPRoute` per service | Generated by SSP AI agent → in tenant namespace |

### Helm chart values

The chart's previous `ingress:` block is removed. The new `route:` block:

```yaml
route:
  enabled: true
  host: hello.acme.ssp.mightybee.dev
  vpnInternal: true   # → attaches to alb-internal-shared
  tls: false
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
```

### TLS

cert-manager is configured with `--enable-gateway-api`. Each `Gateway` (or `HTTPRoute`,
depending on solver config) is annotated with `cert-manager.io/cluster-issuer` and gets a
certificate provisioned automatically. ACME HTTP01 solver is wired against the same
shared Gateway.

## Consequences

- Cost: 2 ALBs total (~$32/mo idle), not one-per-tenant.
- Per-tenant traffic isolation still holds — NetworkPolicy denies cross-namespace pod
  traffic regardless of who attaches to the Gateway.
- If a tenant abuses the shared ALB (cert exhaustion, host hijacking), we can revoke their
  HTTPRoute by removing the tenant label or deleting the route via ArgoCD. Mitigations are
  policy-level, not topology-level.
- AWS LB Controller v2.13+ is a hard floor. The chart pin in `40-platform-addons` is
  `1.13.0`+ — if you bump the controller below that, Gateway API resources will be
  ignored silently.
